FILE: 24_MIDDLEWARE_PIPELINE.txt
================================================================================

REQUEST PIPELINE ORDER:

Every request must pass through middleware in EXACT order:

1. requestId middleware
   → attach unique request ID

2. logger middleware
   → log request metadata

3. security middleware
   → helmet, sanitization

4. cors middleware
   → allow frontend origin only

5. rate limiter middleware
   → prevent abuse

6. auth middleware (if protected route)
   → verify JWT

7. role middleware (if admin route)
   → check permissions

8. validator middleware
   → validate body/query params

9. controller execution

10. response formatter middleware

RULE:
If any middleware fails → stop execution immediately.

FILE: 25_AUTH_FLOW_ENGINE.txt
================================================================================

AUTHENTICATION FLOW:

REGISTER:
1. Validate input
2. Hash password (bcrypt)
3. Create user
4. Generate email verification token
5. Send email via Resend
6. Return success response

LOGIN:
1. Validate input
2. Find user
3. Compare password
4. Check if verified
5. Generate access token (JWT)
6. Generate refresh token
7. Store refresh token in DB
8. Set HTTP-only cookie
9. Return user data

REFRESH TOKEN:
1. Validate refresh token
2. Verify DB match
3. Issue new access token

LOGOUT:
1. Remove refresh token from DB
2. Clear cookies

RULE:
Never expose password or tokens in responses.

SOCKET.IO ARCHITECTURE:

You will implement a namespace-based system:

Namespaces:

/chat
/notifications
/presence
/ai

ROOM STRUCTURE:

chat:{chatId}
user:{userId}

FLOW:

1. User connects
2. JWT verified on socket handshake
3. User joins personal room
4. User joins active chats
5. Presence updated

SCALING RULE:
All socket events must be stateless.

EVENT ROUTING:

Client → Socket Gateway → Event Handler → Service → DB → Emit update

REAL-TIME VOICE & VIDEO SYSTEM:

You will implement WebRTC with Socket.IO signaling.

COMPONENTS:

- peer connection
- ICE candidates
- STUN/TURN servers (free STUN first)
- signaling server (Socket.IO)

FEATURES:

- one-to-one calls
- group calls (future extension)
- screen sharing
- mute/unmute
- camera toggle

FLOW:

1. User initiates call
2. Signal sent via Socket.IO
3. Peer connection established
4. Media streams exchanged

RULE:
No media passes through backend after connection is established.

FILE: 28_AI_ORCHESTRATION_ENGINE.txt
================================================================================

AI SYSTEM ARCHITECTURE:

You will NOT call AI APIs directly from controllers.

Instead:

Controller → AI Service → Provider Router → Model → Response Formatter

PROVIDERS:

1. Gemini (primary)
2. Groq (fast inference)
3. Ollama (local fallback)

ROUTING RULE:

IF task = fast response → Groq
IF task = reasoning → Gemini
IF offline → Ollama

FEATURES:

- chat assistant
- summarization
- OCR
- translation
- code generation
- document analysis
